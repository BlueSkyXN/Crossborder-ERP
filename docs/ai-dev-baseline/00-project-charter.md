# 00 项目定位与净室边界

## 项目一句话定位

构建一个独立实现的跨境代购与集运 ERP，第一版优先跑通集运主链路，保留最小代购能力，后续逐步扩展自动采购、在线支付、PDA、推广返利、多语言和高级财务。

## 不是做什么

本项目不是复刻某个旧系统，也不是把截图里的页面逐像素重做。旧系统和公开材料只作为业务观察来源。

不能做：

- 不复制旧系统 UI 视觉、图标、页面布局、图片素材和原文文案。
- 不复制旧系统私有接口、URL、参数、数据库字段、部署结构。
- 不接触、绕过或使用任何未授权代码、接口、数据和密钥。
- 不把截图中不完整、不确定的信息当作最终业务规则。
- 不在第一版追求完整 SaaS、完整商城、完整自动采购、完整 PDA 和复杂 BI。

可以做：

- 抽象公开可见的业务流程。
- 抽象通用领域对象，例如用户、包裹、运单、钱包、代购订单。
- 抽象通用状态机，例如预报、入库、在库、待付款、已发货。
- 使用独立 UI、独立 API、独立数据库结构重新实现。
- 参考开源 ERP、WMS、TMS 项目的通用工程模式，但要核对授权边界。

## 第一版产品目标

第一版目标不是“功能最多”，而是形成可真实演示、可继续迭代的闭环系统。

必须能演示：

- 后台配置仓库、渠道、包装方式、增值服务。
- 用户注册/登录后获取专属仓库地址。
- 用户提交包裹预报。
- 后台扫描入库，录入包裹重量、体积、图片。
- 用户查看包裹并申请打包发货。
- 后台审核运单和费用。
- 用户使用余额支付。
- 后台发货并录入轨迹。
- 用户查看物流轨迹并确认签收。
- 用户提交最小代购单，后台审核采购，到货后转入包裹链路。

## 默认业务决策

这些是 v0.1 默认决策，除非项目负责人明确修改，否则后续 Agent 按此执行。

| 决策项 | v0.1 默认值 | 说明 |
| --- | --- | --- |
| 第一版主业务 | 集运优先 | 代购做最小能力，不先做自动抓取 |
| 商业形态 | 单商户自用系统 | 不做多租户 SaaS 架构 |
| 首发语言 | 中文 | 数据模型预留多语言字段，但第一版不完整覆盖 |
| 首发移动端 | H5 | 不同时做 App、小程序、原生 |
| 支付 | 余额支付 + 后台人工充值 | 在线支付第二阶段接入 |
| 物流轨迹 | 后台人工录入 | 第三方物流接口第二阶段接入 |
| PDA/硬件 | 页面模拟扫码入库 | 深度接电子秤、打印机、PDA 后置 |
| 自动采购 | 不做 | 淘宝/1688/拼多多抓取与状态同步后置 |
| 推广返利 | 不做 | 运营增长模块后置 |
| 架构 | 模块化单体 | 不上微服务 |

## 成功标准

项目 v0.1 只有同时满足下面条件才算完成：

- 代码可以本地一键启动。
- 数据库迁移和种子数据可以重复执行。
- 后台、用户 Web、移动 H5 至少能跑通核心流程。
- API 有统一鉴权、错误格式、分页和状态枚举。
- 钱包余额变化和流水事务一致。
- 包裹、运单、代购订单状态只能按状态机变化。
- 关键后台操作写入审计日志。
- 有自动化测试覆盖核心后端服务和关键 API。
- 有 README、部署说明、测试账号和演示脚本。

## 开发原则

- 需求驱动：每个代码任务必须能对应到一个需求、流程或验收点。
- 目标导向：先跑通端到端闭环，再补体验和效率功能。
- 小步提交：每个 Agent 任务只改一组相关文件，避免一次性大改。
- 先查后写：遇到已有代码、已有接口、已有模型，先查询再修改。
- 可验证：每个任务都要给出本地验证命令或手工验收步骤。
- 不脑补：不确定字段和业务规则写入 `TODO_CONFIRM`，不得直接固化。
