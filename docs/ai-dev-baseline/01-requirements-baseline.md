# 01 需求基线与 MVP 功能冻结

## MVP 范围总原则

第一版只解决两个问题：

1. 集运主链路完整跑通。
2. 最小代购链路能接入集运。

其他功能只要不影响这两个目标，就延后。

## P0 必做功能

| 业务域 | P0 功能 | 端 | 验收标准 |
| --- | --- | --- | --- |
| 账号与权限 | 用户登录、后台管理员登录、角色权限、菜单权限 | 后台、用户 Web、移动 H5 | 未登录不能访问个人数据；不同后台角色菜单不同 |
| 会员账号 | 会员列表、详情、冻结、重置密码、会员收件标识 | 后台、用户端 | 后台能查会员；用户能看到自己的收件标识 |
| 仓库配置 | 仓库、仓库地址、服务国家/地区、渠道关联 | 后台 | 能配置仓库并在用户端展示专属地址 |
| 基础配置 | 国家地区、包装方式、增值服务、发货渠道 | 后台 | 运单创建和费用计算能引用这些配置 |
| 包裹预报 | 用户提交单条包裹预报 | 用户 Web、移动 H5 | 预报后后台待入库列表可见 |
| 扫描入库 | 后台按快递单号/预报号入库，录入重量、体积、图片 | 后台 | 入库后用户端包裹变为在库 |
| 包裹管理 | 用户查看未入库、在库、已申请打包、已出库包裹 | 三端 | 包裹状态和后台一致 |
| 无主包裹 | 后台登记无主包裹，用户提交认领申请 | 后台、用户端 | 第一版可做简版：后台手动审核归属 |
| 申请打包发货 | 用户选择在库包裹生成运单申请 | 用户 Web、移动 H5 | 已申请包裹不能重复申请 |
| 运单管理 | 待审核、待打包、待付款、待发货、已发货、已签收、已取消、问题单 | 三端 | 运单状态按状态机流转 |
| 运费计算 | 按渠道、重量、体积重、包装、增值服务生成费用 | 后台、用户端 | 后台可调整费用；用户看到费用明细 |
| 钱包余额 | 后台人工充值/扣减，用户余额支付 | 后台、用户端 | 余额变动必须有流水，支付必须幂等 |
| 物流轨迹 | 后台添加轨迹节点，用户查询 | 三端 | 节点按时间展示 |
| 最小代购 | 手工代购/自营商品下单、后台审核采购、到货转包裹 | 三端 | 代购完成后可以生成或关联 Parcel |
| 文件上传 | 商品图、包裹图、凭证图 | 三端 | 文件类型和大小受控，能关联业务单据 |
| 审计日志 | 后台关键操作记录 operator、action、before、after、time | 后台 | 状态、金额、权限、删除类操作均有日志 |

## P1 第二阶段

| 业务域 | P1 功能 | 延后原因 |
| --- | --- | --- |
| 在线支付 | 支付宝、微信、PayPal、信用卡 | 需要第三方账号、回调、安全校验 |
| 线下汇款 | 用户提交汇款单，财务审核入账 | 第一版可后台人工充值替代 |
| Excel 批量预报 | 批量导入包裹预报 | 初期订单量不大，先做单条 |
| 批次发货 | 批量出库、批次物流状态 | 主链路跑通后提高仓库效率 |
| 打印模板 | 面单、标签、拣货单、发票 | 依赖实际打印机和模板确认 |
| 多语言 | 中英语言包完整覆盖 | 第一版先中文，架构预留 |
| 客服消息 | 工单、会话、留言回复 | 可先用简单留言和后台备注 |
| 商品运营增强 | 多语言详情、复杂推荐位、品牌 | 不影响代购和集运闭环 |
| 运费规则增强 | 分区、首重续重、阶梯价、特殊渠道 | 第一版先做可配置基础公式 |

## P2 延后

| 业务域 | P2 功能 | 延后原因 |
| --- | --- | --- |
| 自动采购 | 淘宝/1688/拼多多自动下单、状态同步 | 外部接口、授权、风控复杂 |
| 商品抓取 | 粘贴链接自动解析商品 | 外部依赖复杂，先做手工录入 |
| PDA 深度集成 | PDA App、电子秤、打印机、扫码枪深度对接 | 需要硬件和现场流程 |
| 推广返利 | 邀请、好友、返利流水、海报 | 增长模块，不影响主链路 |
| 拼邮/团长 | 拼团、团长审核、分摊结算 | 状态和财务规则复杂 |
| 多租户 SaaS | 多商户、租户隔离、套餐 | 第一版按单商户系统做 |
| 高级 BI | 利润、成本、渠道、会员高级报表 | 先有真实数据再做 |
| 应用市场 | 插件市场、付费开通 | 平台化能力后置 |

## 明确不做

第一版不要做这些事：

- 不复刻旧 UI。
- 不接真实淘宝/1688/拼多多接口。
- 不做复杂微服务。
- 不做多租户。
- 不做自动采购。
- 不做真实在线支付回调。
- 不做复杂财务应收应付和供应商结算。
- 不做高级仓储盘点和库位优化。

## 核心流程验收

### 流程 1：用户注册和登录

验收：

- 用户可注册或由后台创建。
- 用户可登录、退出、刷新登录态。
- 用户只能访问自己的包裹、运单、钱包流水。
- 后台管理员和普通用户鉴权域分离。

### 流程 2：包裹预报到入库

验收：

- 用户提交预报后生成 `Parcel`，状态为 `PENDING_INBOUND`。
- 后台可按国内快递单号搜索待入库包裹。
- 后台入库后状态变为 `IN_STOCK`。
- 入库重量、体积、图片、备注在用户端可见。

### 流程 3：打包发货到签收

验收：

- 用户只能选择 `IN_STOCK` 包裹申请打包。
- 生成 `Waybill` 后，包裹状态变为 `PACKING_REQUESTED`。
- 后台审核并生成费用，运单进入 `PENDING_PAYMENT`。
- 用户余额支付后，运单进入 `PENDING_SHIPMENT`。
- 后台发货后，运单进入 `SHIPPED`。
- 用户确认收货后，运单进入 `SIGNED`。

### 流程 4：钱包支付

验收：

- 后台人工充值必须生成 `WalletTransaction`。
- 用户支付运单必须生成 `PaymentOrder` 和扣款流水。
- 重复点击支付不会重复扣款。
- 余额不足时不能支付，返回统一错误码。

### 流程 5：最小代购

验收：

- 用户可提交手工代购单或自营商品订单。
- 后台可审核、改价、标记采购中、已采购、到货。
- 到货后可生成或关联 `Parcel`。
- 后续发货复用集运主链路。

## 待确认清单

这些问题不能由 Agent 脑补为最终规则：

| 问题 | 影响 | 默认处理 |
| --- | --- | --- |
| 注册字段、验证码方式 | 用户账号 | 先做账号密码注册，预留手机号/邮箱 |
| 计费公式细节 | 运费计算 | 先做基础可配置公式，后台可人工调整 |
| 首发国家和渠道 | 仓库/渠道 | 种子数据用测试国家和测试渠道 |
| 是否必须中英双语 | UI/数据模型 | 数据预留语言字段，界面先中文 |
| 汇款审核流程 | 财务 | 第一版先后台人工充值 |
| 无主包裹认领凭证 | 异常包裹 | 第一版用户提交说明，后台人工审核 |
| 运单修改范围 | 运单状态机 | 仅 `PENDING_REVIEW` 可改地址、渠道、备注 |
| 打印模板格式 | 仓库出库 | 第一版不做真实打印模板 |
