# 08 测试、安全、部署与交付验收

## 测试金字塔

第一版至少包含：

| 层级 | 工具 | 覆盖范围 |
| --- | --- | --- |
| 后端单元测试 | pytest | service、状态机、计费、钱包 |
| API 集成测试 | DRF APIClient | 登录、包裹、运单、支付 |
| 前端单元测试 | Vitest | 工具函数、状态映射、关键组件 |
| E2E 测试 | Playwright | 端到端主流程 |
| 手工验收 | 浏览器 | 三端核心业务流程 |

## 后端必须测试的内容

### 权限

- 未登录访问用户接口返回 `UNAUTHORIZED`。
- 用户 A 不能访问用户 B 的包裹、运单、钱包。
- 普通后台角色不能操作无权限接口。
- 只隐藏按钮不够，接口也必须鉴权。

### 包裹

- 用户提交预报成功。
- 缺少快递单号失败。
- 重复快递单号处理符合规则。
- 后台入库后状态变为 `IN_STOCK`。
- 非 `IN_STOCK` 包裹不能申请打包。

### 运单

- 创建运单后包裹状态变为 `PACKING_REQUESTED`。
- 状态只能按状态机流转。
- 待付款前费用可调整，付款后费用冻结。
- 已发货后可以添加轨迹。
- 已签收后不能修改。

### 财务

- 人工充值生成流水。
- 余额支付扣款正确。
- 余额不足不能支付。
- 重复支付不会重复扣款。
- 支付失败不改变业务状态。
- 所有金额使用 decimal。

### 代购

- 用户提交手工代购成功。
- 后台审核后状态变化正确。
- 异常单可以恢复或取消。
- 到货后能转成 Parcel。

## 前端必须测试的内容

- 登录失效后跳转登录页。
- 列表 loading/empty/error 都可见。
- 表单必填校验。
- 状态按钮按状态展示。
- 支付按钮防重复点击。
- 移动端底部栏不遮挡提交按钮。
- 金额、重量、体积格式统一。

## E2E 主流程

Playwright 至少覆盖：

```text
后台登录
  -> 创建/确认仓库配置
  -> 用户登录
  -> 复制仓库地址
  -> 用户提交包裹预报
  -> 后台扫描入库
  -> 用户申请打包发货
  -> 后台审核运单并设置费用
  -> 后台给用户充值
  -> 用户支付运单
  -> 后台发货并添加轨迹
  -> 用户查看轨迹并确认收货
```

## 安全要求

第一版即使是测试环境，也要保留基本安全边界：

- 密码必须 hash，不明文存储。
- JWT secret 走环境变量。
- 用户数据必须按 `user_id` 隔离。
- 后台接口必须做 RBAC。
- 文件上传限制类型和大小。
- 图片访问要避免泄露其他用户文件。
- 金额、状态、归属用户不能相信前端传值。
- 后台关键操作写审计日志。
- 支付接口必须幂等。

## 财务一致性要求

钱包和支付是高风险模块，必须满足：

- 所有余额变化由 `WalletTransaction` 产生。
- 扣款和业务状态更新在一个事务内。
- 钱包余额不能为负。
- 退款、扣减、充值都要记录操作人和原因。
- 支付单号唯一。
- 幂等键唯一。

## 部署要求

### 本地部署

必须提供：

- `.env.example`
- `docker-compose.yml`
- 数据库迁移命令
- 种子数据命令
- 三端启动命令

### Staging 部署

必须说明：

- 数据库地址。
- Redis 地址。
- 后端域名。
- 三端前端域名。
- 文件存储位置。
- 如何执行迁移。
- 如何回滚代码。

### 生产部署后续再做

第一版可以不做复杂生产部署，但代码结构要支持：

- 环境变量配置。
- 静态资源构建。
- 媒体文件外置。
- 日志输出。
- 数据库备份。

## 演示脚本

交付时必须能按以下脚本演示：

1. 用超级管理员登录后台。
2. 查看仓库、渠道、包装、增值服务配置。
3. 用测试用户登录用户 Web。
4. 复制仓库地址。
5. 提交包裹预报。
6. 回到后台扫描入库。
7. 用户端查看包裹在库。
8. 用户申请打包发货。
9. 后台审核并设置费用。
10. 后台给用户充值。
11. 用户支付运单。
12. 后台发货并录入轨迹。
13. 用户查看轨迹。
14. 用户确认收货。
15. 提交一个手工代购单。
16. 后台审核采购并到货转包裹。
