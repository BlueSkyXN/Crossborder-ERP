# 13 综合产品规格：最终要做什么

本文件是基于 ChatGPT/Gemini 三端分析、截图和前端复现包综合后的最终产品定义。后续开发以本文件为主，不再把三端分析文档直接当开发输入。

## 产品定位

产品名称暂定：`CrossBorder ERP`。

产品本质：

```text
跨境代购 + 集运转运 + 仓储入库 + 运单发货 + 会员钱包 + 三端入口 的模块化 ERP
```

第一版不是完整行业 SaaS，而是一个单商户可用的端到端系统：

- 后台管理端：运营、仓库、采购、财务、客服使用。
- 用户 Web：会员在电脑端管理代购、包裹、运单、财务。
- 移动 H5：会员在手机端完成高频轻操作。
- 后端 API：三端共用同一领域模型和状态机。

## 产品分层

| 层 | 内容 |
| --- | --- |
| 业务层 | 代购、集运、仓储、运单、财务、会员、商品、内容、客服 |
| 应用层 | 后台管理端、用户 Web、移动 H5 |
| 服务层 | Django/DRF 模块化单体 |
| 数据层 | PostgreSQL、Redis、文件存储 |
| 集成层 | 支付、物流、商品抓取、打印、PDA 后续插件化接入 |

## 三端定位

### 后台管理端

目标：让运营团队处理订单、包裹、运单、采购、财务和配置。

核心能力：

- 管理员登录和 RBAC。
- 基础配置：国家地区、仓库、渠道、包装、增值服务。
- 会员管理：账号、等级、冻结、客服分配、收件标识。
- 商品管理：分类、商品、SKU、图片、推荐位。
- 代购管理：待付款、待审核、待采购、已采购、已完成、异常/取消。
- 包裹仓储：待入库、扫描入库、在库、无主包裹。
- 运单管理：待审核、待打包、待付款、待发货、已发货、已签收、问题单。
- 财务管理：钱包、流水、人工充值、人工扣减、支付单。
- 内容/客服/审计：帮助、公告、留言、操作日志。

后台第一版界面以高密度表格和详情页为主，不做营销型页面。

### 用户 Web

目标：让会员完整管理代购和集运业务。

核心能力：

- 登录/注册。
- 控制台：账户、余额、积分、订单状态、仓库地址、快捷入口。
- 商品/代购：商品搜索、商品列表、详情、购物车、确认订单、万能代购。
- 包裹：预报、列表、详情、申请打包。
- 运单：列表、详情、支付、追踪、确认收货。
- 财务：钱包余额、流水、充值入口。
- 地址：海外收件地址管理。
- 客服/消息：基础留言或入口。

用户 Web 第一版优先跑通业务闭环，不追求完整商城体验。

### 移动 H5

目标：让会员在手机上完成高频动作。

底部五栏：

```text
首页
分类
寄件
购物车
我的
```

核心能力：

- 登录。
- 首页搜索、快捷入口、商品推荐。
- 寄件首页：仓库地址、集运/直邮 Tab、复制地址、发布预报、申请打包。
- 包裹详情和申请打包。
- 运单列表、追踪、确认收货。
- 我的：账户概览、代购订单、转运中心、运单、余额、设置。

移动 H5 第一版必须有触控友好的卡片、Tab、底部操作栏和空状态。

## P0 业务闭环

### 闭环 A：集运主链路

```text
用户登录
  -> 查看/复制仓库地址
  -> 在外部电商下单并获得国内快递单号
  -> 提交包裹预报
  -> 后台待入库列表出现 Parcel
  -> 仓库扫描入库，录入重量/体积/图片
  -> Parcel 变为 IN_STOCK
  -> 用户选择在库包裹申请打包
  -> 创建 Waybill
  -> 后台审核/打包/设置费用
  -> Waybill 变为 PENDING_PAYMENT
  -> 后台给用户充值或用户已有余额
  -> 用户余额支付
  -> Waybill 变为 PENDING_SHIPMENT
  -> 后台发货并录入 TrackingEvent
  -> Waybill 变为 SHIPPED
  -> 用户查看轨迹并确认收货
  -> Waybill 变为 SIGNED
```

### 闭环 B：最小代购链路

```text
用户登录
  -> 提交手工代购或自营商品订单
  -> PurchaseOrder 进入 PENDING_PAYMENT/PENDING_REVIEW
  -> 用户支付或后台确认付款
  -> 后台审核价格和备注
  -> 采购人员处理采购
  -> 采购到货
  -> 后台将到货商品转成 Parcel
  -> Parcel 接入集运主链路
```

### 闭环 C：钱包支付链路

```text
后台人工充值
  -> Wallet 增加
  -> WalletTransaction 记录 ADMIN_RECHARGE
  -> 用户支付 Waybill/PurchaseOrder
  -> PaymentOrder 创建并 PAID
  -> WalletTransaction 记录 WAYBILL_PAYMENT/PURCHASE_PAYMENT
  -> 业务单据进入下一状态
```

## 业务模块总表

| 模块 | P0 功能 | P1 功能 | P2/延后 |
| --- | --- | --- | --- |
| IAM | 用户登录、后台登录、RBAC、菜单权限 | 登录日志、二次验证 | 企业 SSO |
| Member | 会员资料、收件标识、冻结、重置密码 | 会员等级权益 | 成长体系 |
| Config | 仓库、国家地区、渠道、包装、增值服务 | 复杂费率、渠道分区 | 多仓策略优化 |
| Product | 分类、商品、SKU 简版 | 复杂 SKU、推荐位、多语言详情 | 商品抓取和同步 |
| Purchase | 手工代购、采购处理、到货转包裹 | 分批采购、凭证、异常处理增强 | 自动采购 |
| Parcel/WMS | 预报、入库、在库、无主简版 | 批量导入、库位、标签打印 | PDA/电子秤/高拍仪 |
| Waybill/TMS | 申请打包、审核、计费、支付、发货、轨迹 | 批次发货、转单、面单打印 | 物流 API 自动同步 |
| Finance | 钱包、人工充值/扣减、流水、余额支付 | 线下汇款审核、退款 | 应收应付、利润核算 |
| Content | 基础帮助和公告 | CMS 富文本 | 多语言 CMS |
| Ticket | 简单留言入口 | 工单会话 | 第三方客服深度集成 |
| Marketing | 不做 | 积分明细入口 | 推广返利、分销 |
| Report | 简版控制台 | 运营统计 | 高级 BI |

## 不确定项必须保留

这些功能在截图/分析里有入口或推断，但规则不完整，第一版不能硬编码复杂规则：

- 登录注册字段、验证码、找回密码。
- 商品链接解析和图片搜索。
- 复杂 SKU 抓取。
- 计费公式：首重续重、体积重、偏远费、保险费。
- 直邮完整流程。
- 无主包裹认领凭证和审核细节。
- 在线支付回调和币种结算。
- 推广返利比例、积分兑换规则。
- 真实打印模板和硬件接入。

处理原则：

```text
如果 P0 主链路必须用到，就做简化版 + TODO_CONFIRM。
如果不影响 P0 主链路，就放 P1/P2。
```
