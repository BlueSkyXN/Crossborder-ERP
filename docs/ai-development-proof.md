# AI 驱动开发证明口径

本项目除了交付一个跨境代购与集运 ERP，还要证明：一个复杂 ERP 全栈系统可以在极低人工介入下，由 AI Agent 持续推进完成。

## 证明目标

目标不是声称“完全没有人类参与”。本项目的人类参与限定为：

- 提供业务目标、约束和优先级。
- 对关键业务边界作确认，例如数据库验证边界、是否使用 Docker、是否接真实外部接口。
- 对阶段性交付结果做验收或纠偏。

AI Agent 负责：

- 阅读规格和任务图。
- 拆解当前任务。
- 选择实现方案。
- 编写代码、测试、文档和配置。
- 运行验证。
- 记录验证结果、风险和下一步。

## 人工介入边界

允许的人工介入：

- 明确产品目标和业务范围。
- 确认待确认项，例如 `TODO_CONFIRM`。
- 指定约束，例如 no-Docker、SQLite-first、不真实验证 PostgreSQL/MySQL/Redis。
- 验收演示结果。
- 要求回滚、暂停或调整优先级。

不作为常规流程依赖的人工介入：

- 人工设计代码结构。
- 人工手写核心业务代码。
- 人工补测试。
- 人工排查常规构建、语法、路由、依赖问题。
- 人工代替 Agent 阅读文档或判断下一任务。

## 证据要求

证据留存采用“轻量、阶段化、可复核”的原则。不要记录每轮对话、每个小修小补或完整命令日志；只在正式任务 ID 或阶段里程碑完成时留下简明证据。

正式任务完成后，Agent 必须留下可审计摘要：

- 关联任务 ID，例如 `BE-001`。
- 关联需求文档或规格文件。
- 关键 Agent 决策，限核心取舍。
- 修改范围摘要，不逐文件解释细节。
- 验证命令和结果摘要。
- 未验证能力边界。
- 剩余风险和下一步。

通常每个正式任务 ID 最多一份记录。纯文案微调、`.gitignore` 小修、清理 `.DS_Store` 等维护动作，不需要单独写 `agent-runs`，可由 commit message 或最终汇报覆盖。

对于未真实验证的能力，必须使用明确措辞：

```text
configured_unverified
```

例如 PostgreSQL/MySQL/Redis/Celery 当前只做配置兼容时，不得写成“已支持生产部署”或“已完成验证”。

## 任务推进规则

默认按下面顺序推进：

1. 读取 `docs/ai-dev-baseline/agent-execution/current-state.yaml`。
2. 打开当前任务文件。
3. 检查依赖是否完成。
4. 执行一个任务，不扩大范围。
5. 完成正式任务后更新当前状态；必要时补一份简明任务记录。
6. 再进入下一个任务。

人类不需要逐步给出技术细节；Agent 应根据项目规则自主推进。

## 建议记录格式

正式任务或阶段里程碑可以在 `docs/agent-runs/` 下记录一份简短 Markdown：

```text
docs/agent-runs/YYYY-MM-DD-<task-id>.md
```

内容模板：

````markdown
# <task-id> <任务名>

## 输入

- 任务文件：
- 关联规格：
- 人工约束：

## Agent 决策

- 只记录关键方案取舍，不记录完整思考过程。

## 修改

- 修改范围摘要。

## 验证

```bash
<command>
```

结果：

```text
<summary result, not full logs>
```

## 未验证边界

- 

## 下一步

- 
````

建议长度：普通任务控制在 300-800 中文字；复杂任务可以略长，但不要粘贴完整日志、完整 diff 或长篇过程复盘。

## 成功判断

如果最终系统能在以下条件下交付，本项目即可支持“AI 驱动全栈 ERP 开发”的证明：

- 后端、后台、用户 Web、移动 H5 都由 Agent 依据规格逐步实现。
- 核心链路测试和演示脚本由 Agent 维护。
- 主要正式任务或阶段里程碑都有简明 Agent 运行记录。
- 人工介入集中在目标、约束、验收和少量业务确认。
- 未验证能力均有清晰边界，不夸大为已验证结论。
