# 00 产品定位：轻量版 KINGANT-like 跨境代购/集运平台

## 1. 背景

目标参考对象是 KINGANT（金蚁）这类跨境代购与集运转运综合服务平台。该类平台通常包含：

- 用户端：商品代购、万能代购、包裹预报、集运发货、无主包裹认领、运费估算、充值、线下汇款、余额、流水、好友返利、积分、个人资料、地址、消息。
- 后台端：入库、仓储标签、打包、付款运单、待发货、批次发货、转单、问题件、代购审核/采购/到货、应收应付、汇款审核、供应商、商品、会员、内容、基础设置、控制台看板。

但当前项目不应一次性做成重型 ERP。新的目标是：

> 轻量、好用、美观、可跑通核心业务闭环的代购/集运服务平台。

## 2. 目标产品一句话

为跨境用户提供一个简单清晰的平台：

```text
买东西 / 发包裹 -> 国内仓库入库 -> 合箱打包 -> 付运费 -> 国际发货 -> 查轨迹
```

后台则提供：

```text
收货入库 -> 审核计费 -> 财务入账 -> 发货批次 -> 轨迹维护 -> 会员/商品/内容配置
```

## 3. 产品形态

### 3.1 用户端

一个响应式前台，支持桌面和移动端：

- 首页
- 商品/代购入口
- 仓库地址
- 包裹预报
- 我的包裹
- 申请打包/运单
- 财务中心
- 工单/消息
- 我的账户

### 3.2 后台端

同一个前端应用内的后台 `/admin`：

- 控制台
- 包裹入库
- 运单处理
- 发货批次
- 代购订单
- 财务审核
- 会员管理
- 商品管理
- 内容管理
- 基础设置

### 3.3 技术形态

保持现有技术栈：

```text
Django + DRF + SQLite-first + React/Vite/TypeScript + pnpm workspace
```

不拆微服务，不重写后端。

## 4. 轻量化原则

### 4.1 少页面

用户端不超过 10 个核心页面，后台不超过 12 个核心页面。复杂能力通过详情页、抽屉、弹窗或 Tab 承载。

### 4.2 少状态

状态必须覆盖业务，但不要过度细化。

包裹状态建议：

```text
PENDING_INBOUND
IN_STOCK
PACKING_REQUESTED
OUTBOUND
SIGNED
PROBLEM
CANCELLED
```

运单状态建议：

```text
PENDING_REVIEW
PENDING_PACKING
PENDING_PAYMENT
PENDING_SHIPMENT
SHIPPED
SIGNED
CANCELLED
PROBLEM
```

代购状态建议：

```text
PENDING_REVIEW
PENDING_PROCUREMENT
PROCURED
ARRIVED
COMPLETED
CANCELLED
EXCEPTION
```

### 4.3 少外部依赖

P0 继续使用：

- SQLite。
- 本地文件存储。
- 同步任务。
- 余额支付。
- 线下汇款。
- 后台人工充值。
- 手工物流轨迹。

### 4.4 美观但不重

目标是“现代 SaaS 风格”，不是复杂动效：

- 大留白。
- 卡片式布局。
- 明确主按钮。
- 状态标签统一。
- 首页有清晰行动入口。
- 移动端底部导航。
- 后台表格密度适中。

## 5. 明确非目标

P0 不做：

- 自动淘宝/天猫真实抓取。
- 自动下单。
- 微信/支付宝/PayPal/Qpay/信用卡真实网关。
- 物流 API 自动轨迹同步。
- 打印机硬件集成。
- 复杂库位、拣货波次、PDA。
- 复杂会计总账。
- BI 大屏。
- 病毒扫描。
- Docker/K8s。
- 多租户 SaaS。

## 6. 阶段目标

### P0：轻量可演示闭环

- 用户可以注册/登录。
- 用户可以复制仓库地址。
- 用户可以提交包裹预报。
- 后台可以扫码入库。
- 用户可以申请打包并生成运单。
- 后台可以审核计费。
- 用户可以通过余额支付；余额来源为后台充值或线下汇款审核。
- 后台可以发货、追加轨迹。
- 用户可以确认收货。
- 用户可以提交万能代购。
- 后台可以采购到货并转包裹。

### P1：更像 KINGANT 的完整轻量版

- 运费估算。
- 无主包裹认领。
- 发货批次/转单。
- 商品库和购物车代购。
- 好友返利/积分基础。
- 内容 CMS。
- 工单消息。
- 统一前端视觉升级。

### P2：生产增强

- MySQL/PostgreSQL 真实验证。
- 线上支付网关。
- 物流 API。
- 对象存储。
- 监控和备份恢复。
