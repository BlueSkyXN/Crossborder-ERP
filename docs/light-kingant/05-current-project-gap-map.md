# 05 当前项目与轻量 KINGANT-like 目标差距地图

## 1. 当前项目已有能力

当前项目已经具备大量 KINGANT-like 平台所需基础：

- 会员注册、登录、找回密码、账户设置。
- 仓库地址和会员识别码。
- 包裹预报。
- 后台扫描入库。
- 无主包裹认领。
- 运单创建、审核、计费、余额支付、发货、轨迹、确认收货。
- 发货批次、转单号、打印模板数据。
- 万能代购和外部链接解析。
- 商品分类、商品、SKU、购物车。
- 钱包、线下汇款、后台充值、流水。
- 应付基础。
- 工单消息。
- 内容 CMS。
- 积分/推广/返利基础。
- RBAC、审计、安全响应头、CSV 防注入、文件 sniff。
- API E2E 和浏览器 smoke。

因此不需要推倒重写。

## 2. 当前项目偏重的问题

项目当前文档和功能有“重型 ERP 化”倾向：

- 生产化 backlog 很长。
- Provider 抽象较多。
- 外部服务边界较多。
- 三端前端维护成本高。
- 部分后台页面过大。
- 文档定位和实际轻量产品目标不完全一致。

## 3. 与 KINGANT-like 目标的主要差距

| 领域 | 当前状态 | 差距 | 优先级 |
| --- | --- | --- | --- |
| 产品定位 | 跨境代购与集运 ERP | 需要收敛为轻量代购/集运平台 | P0 |
| 统一前端 | admin-web/user-web/mobile-h5 三端 | 需要一个美观统一响应式 web-app | P0 |
| 用户首页 | 已有入口但产品感不足 | 需要流程化、美观、清晰行动入口 | P0 |
| 仓库地址 | 已有 | 需要更强的一键复制和新手说明 | P0 |
| 包裹预报 | 已有 | 需要统一前端重做体验 | P0 |
| 运单流程 | 已有 | 需要统一前端重做体验 | P0 |
| 万能代购 | 已有基础 | 需要更像用户提交代购需求的轻量页面 | P0 |
| 后台入库 | 已有 | 需要更高效的扫码输入 UX | P0 |
| 后台运单处理 | 已有 | 需要拆大页面、状态 Tab 化 | P0 |
| 财务 | 已有线下汇款/钱包 | 多渠道真实支付后置 | P1/P2 |
| 运费估算 | 已有基础 | 需要首页/用户端突出展示 | P1 |
| 推广积分 | 已有基础 | 规则需轻量化，不做复杂分销 | P1 |
| 商品代购 | 有商品和购物车 | 需要明确其是代购入口而非普通电商 | P1 |
| 打印/转单 | 已有模板数据 | 真实硬件和物流 API 后置 | P2 |
| 数据大屏 | 有 dashboard | 不做复杂 BI，保持轻量 | P1 |

## 4. 保留但降级的能力

这些能力保留，但不是当前主线：

- 对象存储。
- 真实物流 API。
- 真实在线支付。
- Redis/Celery。
- MySQL/PostgreSQL 真实验证。
- 复杂应付/结算。
- 复杂权限审批。
- 外部 SIEM。
- 病毒扫描。

## 5. 需要删改的旧口径

文档中应避免继续把项目描述为：

```text
完整生产级 ERP
完整财务系统
完整 WMS
完整对象存储/病毒扫描体系
真实线上支付完成
真实物流 API 完成
```

应改为：

```text
轻量跨境代购/集运平台
SQLite-first 本地可验证
P0 主链路完成
生产化能力逐步验证
```

## 6. 最小成功标准

如果以下流程能在统一前端中跑通，即可视为轻量 KINGANT-like MVP 成功：

```text
用户端：注册/登录 -> 仓库地址 -> 包裹预报 -> 我的包裹 -> 申请打包 -> 运单支付 -> 查看轨迹
后台端：登录 -> 入库 -> 运单审核计费 -> 汇款审核 -> 发货 -> 审计日志
代购：用户提交万能代购 -> 后台采购到货 -> 转包裹
```

## 7. 后续是否再回到“发卡平台”

如果业务后来确认需要在线发卡，可以作为单独模块加入：

```text
digital_goods
card_stock
digital_delivery
```

但根据 KINGANT-like 目标，发卡不应作为当前主线。
