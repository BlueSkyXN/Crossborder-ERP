# 下一阶段目标与范围

本文件定义下一阶段“为什么做、做到什么程度、哪些不做”。后续 Agent 必须先用本文件判断任务是否符合目标，再进入实现。

## 阶段定位

当前系统不是从零开始。下一阶段的目标是把已经完成的 SQLite-first ERP MVP，推进到“可试运行、可验证、可持续由 Agent 维护”的工程状态。

一句话目标：

```text
从 P0 SQLite-first main flows complete，推进到 staging/sandbox ready 的可试运行基线。
```

这里的 `staging/sandbox ready` 不是生产承诺，而是表示：

- 真实生产前硬阻塞项有明确验收路径。
- 能力状态用 `not_implemented`、`configured_unverified`、`local_verified`、`sandbox_verified`、`production_verified` 区分。
- 外部依赖通过 provider 抽象接入，不散落硬编码。
- 后续 Agent 能按任务卡、规格、设计和验证矩阵继续推进。

## 北极星目标

| ID | 目标 | 成功标准 |
| --- | --- | --- |
| GOAL-01 | 建立下一阶段 Agent 控制面 | 后续 Agent 可从 `docs/next-phase/` 选择任务、理解边界、执行验证 |
| GOAL-02 | 保持主链路稳定 | 集运主链路、最小代购、钱包、三端浏览器 smoke 不因生产化改造退化 |
| GOAL-03 | 消除试运行硬阻塞 | production settings、认证安全、数据持久化、备份恢复、对象存储或 media 持久化、监控告警有可验证路径 |
| GOAL-04 | 外部系统 provider 化 | 支付、物流、通知、对象存储、病毒扫描、自动采购都先有 disabled/local/fake provider 和明确边界 |
| GOAL-05 | 建立真实验证分层 | SQLite、本地 fake、sandbox、staging、production 的证据分开记录，不混用口径 |
| GOAL-06 | 需求规格驱动开发 | 每个任务关联需求 ID、设计决策、验收标准、未验证边界和证据记录 |

## 范围分层

### 第一层：下一阶段必须先做

这些任务用于让后续 AI 能正确开发，优先级最高：

- 任务控制文档和任务卡规范。
- 需求规格、目标架构、验证矩阵和人工确认清单。
- production settings 安全硬化。
- 认证安全硬化，包括登录限流、密码策略、生产禁用 demo token 暴露。
- 对象存储和病毒扫描 provider 抽象，默认仍可使用 local provider。
- 结构化日志、readiness 扩展和监控告警文档。
- PostgreSQL 真实验证计划和可执行任务卡。

### 第二层：具备条件后推进

这些任务需要环境、账号或业务确认：

- PostgreSQL 真实连接、迁移、pytest、钱包并发验证。
- Redis/Celery 真实 broker、重试、失败补偿验证。
- staging 部署、TLS/HSTS、Nginx 反代、回滚脚本。
- 支付 sandbox provider。
- 物流 sandbox provider。
- 短信、邮件、微信通知 provider。
- 真实对象存储、签名 URL、CDN、生命周期、备份。
- Playwright、视觉回归、性能基线、安全测试。

### 第三层：暂不作为默认任务

这些事项不应由 Agent 未经确认自行启动：

- 微服务化。
- 多租户 SaaS。
- 真实线上支付生产接入。
- 真实自动采购下单。
- 第三方平台账号密码保存。
- 旧系统 UI 复刻。
- 复杂 BI、大屏、财务税务自动化。
- PDA、电子秤、打印机深度硬件集成。
- Docker 化，除非用户重新确认当前阶段允许。

## 非目标

下一阶段不是：

- 不是把所有源报告功能一次性补满。
- 不是把前端重做一遍。
- 不是把本地 SQLite 版本直接上线。
- 不是引入微服务、消息总线、Kubernetes 等额外复杂度。
- 不是让 AI 自己补齐业务规则。
- 不是以“文档齐全”为完成，文档必须服务于后续可执行任务。

## 试运行前硬阻塞项

真实用户、真实资金或真实包裹试运行前，以下事项至少要有验证证据或明确人工豁免：

| 阻塞项 | 最低完成口径 |
| --- | --- |
| 数据库 | PostgreSQL 真实连接、迁移、测试和关键并发场景验证 |
| 配置 | production settings 强制环境变量、关闭 debug、关闭 demo token 暴露 |
| 认证 | 登录限流、密码策略、旧 token 失效、后台高风险操作保护 |
| 文件 | 对象存储或可持久化 media 方案、备份、访问控制、病毒扫描或受控替代方案 |
| 备份 | 数据库和文件恢复演练，记录 RTO/RPO 边界 |
| 部署 | staging、TLS、反向代理、readiness、日志和回滚步骤 |
| 钱包 | 并发扣款、幂等、防重复支付在目标数据库下验证 |
| 审计 | 高风险操作审计、导出审批或最小替代流程 |
| 业务规则 | 运费、赔付、禁运、退款、审批、供应商付款等核心规则人工确认 |

## 成功判断

下一阶段不能只看代码数量。判断标准是：

- 主链路回归仍可运行。
- 新能力都有需求 ID 和验收标准。
- 未验证能力没有被包装成已完成。
- 每个外部依赖都有 provider 边界。
- 后续 Agent 不需要猜下一步。
- 人工确认项被集中记录，没有散落在代码注释里。
